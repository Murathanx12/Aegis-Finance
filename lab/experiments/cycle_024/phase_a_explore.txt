### Finding 2: 13 broad `except` blocks silently swallow bugs (code smell)

**`stock_analyzer.py`** — 8 bare `except Exception: return None` blocks at lines 312, 335, 372, 392, 428, 442, 487, 496. Every helper (`_get_analyst_targets`, `_get_recommendations`, `_get_holders`, `_get_news`, `_get_earnings`, `_get_key_stats`, `_get_sector_peers`) catches ALL exceptions and returns `None`. A `TypeError` from a yfinance API change would be invisible.

**`portfolio_engine.py`** — 5 broad excepts at lines 341, 569, 572, 664, 667. The BL and HRP optimization paths (`_build_bl`, `_build_hrp`) have nested try/except that swallow optimization failures silently, then fall back to template without logging what actually failed.

**Fix**: Narrow these to expected exception types (`ValueError`, `KeyError`, `AttributeError` for yfinance parsing; `np.linalg.LinAlgError` for optimization), and log the swallowed exceptions at `logger.debug` level.

---

### Finding 3: `regime_validator._check_consensus` uses hardcoded 3% threshold against stale config

**File**: `backend/services/regime_validator.py:160-184`

The consensus check reads `config["institutional_benchmarks"]` and compares the mean annual return against a hardcoded 3% threshold (line 179). Two issues:

1. **Threshold not in config** — The 3% bear/bull cutoff is hardcoded, violating the project rule "Put all parameters in `backend/config.py`"
2. **Stale benchmarks** — If institutional benchmarks in config haven't been updated, this check gives false consensus alignment. With current benchmarks likely showing ~5.9% annual return (from CLAUDE.md), `_check_consensus` will ALWAYS return `True` for bull and `False` for bear, making it a rubber stamp rather than a real check.

This means the regime validator's "3 independent checks" are really just 2 — the consensus check provides no signal.
