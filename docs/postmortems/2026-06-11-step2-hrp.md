## THE BIG FIND — rebalance events never traded the book

`run_reference_check` recorded rebalance events (pre/post weights, trades,
costs) but **never rewrote `paper_positions`** — positions were only ever
created at `initialize_lane`. No natural rebalance had fired yet (monthly
cadence, day 2-3), so the live record is uncontaminated — but the first
monthly rebalance (~2026-07-08) would have started silently diverging the
recorded weights from the NAV-driving book, and the v1→v2 "boundary" would
have been an event with no allocation change. Fixed:
`_apply_rebalance_positions` closes the open book and re-opens at target
weights / current prices, net of transaction costs. Test-pinned: booked
weights ≈ post_weights (1e-4), NAV continuous across the switch (≤ costs),
rebalances cannot create value.

## Walk-forward sanity (gate c — NOT a skill claim)

Replay 2021-01-04 → 2025-12-31, balanced mandate, crash override 0.0 (overlay
off), identical universe/costs — only the optimizer differs:

| | HRP (balanced) | Equal-weight (control) |
|---|---|---|
| Total return | 68.8% | 125.7% |
| Ann. return | 11.1% | 17.8% |
| Ann. vol | **10.9%** | 14.9% |

| Cost | 58.5 bps | 36.5 bps |

**Verdict on the gate: NOT degenerate** — HRP does exactly what HRP does:
clusters risk toward low-vol names → 27% less volatility, 7pts shallower
max drawdown, and lower raw return in a window (2021–25) that paid high-beta
growth, at higher turnover. **Honest finding: HRP loses on backtest Sharpe
over this window.** We are switching anyway and that is the correct,
pre-registered move: the window is one survivorship-flagged draw with a
stubbed crash model, the lane mandate is risk-managed (lower DD is mandate-
consistent), and "does HRP add value forward" is precisely the registered
hypothesis the ew-control lane now measures. If we only shipped optimizers
that win backtests, the registry would be an overfitting machine. If forward
data agrees with the backtest, the evolution loop reverts this as config v3
— with the receipts.

## Rollback note (do not revert in place)

The config hash is content-based: reverting the YAML reproduces the v1 hash
exactly, making "v1" non-contiguous in the record. Rollback = roll FORWARD
(e.g. set `optimizer: equal_weight` AND add a `config_note: rollback-...`
field) so a fresh hash starts a fresh segment.
